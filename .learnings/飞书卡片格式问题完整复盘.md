# 飞书卡片格式问题完整复盘（2026-04-10）

## 问题现象
在飞书收到的是原始JSON字符串，而非可交互的卡片：
```
{"schema": "2.0", "header": {"title": {"tag": "plain_text", "content": "..."}}, "body": {"elements": [...]}}
```

## 根因分析

### 错误认知
一直以为问题出在JSON格式、schema版本、tag类型

### 正确认知
问题是**参数传递位置错误**

```
❌ 错误：message参数 → 飞书收到的是文本
message: "{\"schema\": \"2.0\", ...整个JSON...}"

✅ 正确：card参数 → 飞书收到的是卡片
message: "请查看卡片"
card: {"schema": "2.0", ...}
```

## 经验教训

### 1. 调试问题时先确认工具参数结构
- 不应该花大量时间调整JSON格式

- `card`参数在OpenClaw代码里确实存在

### 2. 遵循最小化复现原则
- 问题一开始就去检查`card`参数，不至于花1小时
- 应该创建最小测试用例验证假设

### 3. 看日志找线索
- 日志里有400错误，但被忽略了
- 应该追踪为什么返回400

### 4. 配置变更要完整
- 添加了`renderMode: card`但不知道还要改参数传递方式
- 新配置需要完整测试，不能只测一半

## 正确的飞书卡片发送流程

### Step 1: 配置OpenClaw（一次性）
```
openclaw.json → channels.feishu.renderMode: "card"
```

### Step 2: 发送时正确使用参数
```
message工具 action=send
- message: "请查看卡片"（备用文本）
- card: {schema: "2.0", header: {...}, body: {...}}
```

## 关键结论

**飞书卡片发送 = renderMode配置 + card参数传递**

两个条件缺一不可：
1. `channels.feishu.renderMode: "card"` - 让OpenClaw知道要用卡片模式
2. `card参数` - 让OpenClaw知道把JSON当卡片发而不是当文本发
